# ClipVault Personal v1.5 — CandidateMixer

Date: 2026-06-21

Scope:

> unify Android IME candidate generation into a deterministic, testable runtime mixer.

This version is not about adding Chinese input or ML ranking. It is the architecture step before those larger changes.

## 1. Research inputs

### 1.1 Privacy keyboards point to thin frontends and explicit learning controls

References reviewed:

- HeliBoard / OpenBoard: https://github.com/Helium314/HeliBoard and https://github.com/openboard-team/openboard
- FlorisBoard: https://github.com/florisboard/florisboard
- AnySoftKeyboard: https://github.com/AnySoftKeyboard/AnySoftKeyboard

Takeaway:

- Keyboard UI should be thin; candidate generation should be behind a runtime/service layer.
- Candidate learning must be explicit and controllable.
- GPL keyboard projects are useful for architecture and UX reference, but should not be copied into this repository without a licensing decision.

Decision:

- Add CandidateMixer inside the ClipVault runtime layer.
- Keep Panel/FullKeyboard as frontends that only render runtime candidates.

### 1.2 Smart replies / advanced suggestions should stay local

Reference:

- Wired on Gboard Smart Reply privacy: https://www.wired.com/story/gboard-smart-reply-privacy

Takeaway:

- Smart keyboard suggestions create privacy risk because they may analyze the user's active context.
- The relevant lesson for ClipVault is not to copy Gboard features, but to keep ranking deterministic, local, and non-telemetric.

Decision:

- v1.5 CandidateMixer uses deterministic scores only.
- No network ranking.
- No typed-text learning.
- No LLM candidate generation.

### 1.3 Keyboard-logging research reinforces the no-keystroke boundary

References:

- WildKey privacy-aware keyboard toolkit: https://arxiv.org/abs/2105.10223
- Smartphone keyboard logging/context study: https://arxiv.org/abs/2403.05180
- Hover attack on Android input: https://arxiv.org/abs/1611.01477

Takeaway:

- Keyboard input data is sensitive even when collected for research or personalization.
- ClipVault's value should come from explicit clips and Personal Memory, not ambient typed text.

Decision:

- CandidateMixer consumes only existing stored clips/memory.
- It does not persist ordinary typed text.

### 1.4 Public-platform search notes

The targeted search pass covered GitHub, Hacker News, Reddit, Medium, LinkedIn, Lobsters, Stack Overflow, and broader web search terms around Android IME candidate strips and privacy keyboards. Direct, reusable project ideas beyond the open-source keyboard references above were sparse; the useful implementation pattern was architectural rather than code-level:

- central candidate model
- thin UI renderer
- deterministic local scoring
- sensitive-field guard before ranking

## 2. Implementation decisions

### D-019 Candidate model

Runtime candidate shape:

- `id`
- `source`: `clip` or `memory`
- `kind`: clip content type or memory kind
- `text`
- `label`
- `score`
- `riskFlags`

### D-020 CandidateMixer scoring

Deterministic score components:

- memory base > clip base, because explicit Personal Memory is more intentional than raw history
- pinned boost
- prefix match boost
- favorite/use-count/times-seen boost
- kind weights for phrase / prompt / command / term / key_info / path
- stable tie-breakers: score desc, source, kind, label, id

### D-021 PrivacyAwareFilter remains before mixer display

`PrivacyAwareFilter` is still the IME-side first gate. In sensitive fields, frontends should not call into candidate display at all.

### D-022 Compatibility API retained

The old `listRecentClips()` and `listMemory()` APIs are retained for compatibility. New frontends should use `listCandidates()`.
